# RAG / Голосовой поиск — настройка Firestore Vector Search

Этот документ описывает, что нужно сделать в инфраструктуре (Firebase/Firestore),
чтобы заработал голосовой/текстовый RAG-поиск по записям (`POST /api/ai/search`).
Сам код (чанкинг, эмбеддинги, индексация, поиск) уже реализован и НЕ требует
доработки — но векторный индекс в Firestore нужно создать руками один раз.

## 1. Модель эмбеддингов

- Модель: `text-embedding-004` (Gemini Embedding API, тот же ключ `GEMINI_API_KEY`,
  что и для всех остальных AI-вызовов — **новые env-переменные не нужны**).
- Размерность вектора: **768** (константа `EMBED_DIM` в `server/lib/embeddings.ts`).
- Коллекция чанков: `users/{uid}/chunks/{chunkId}`, поле с вектором — `embedding`
  (тип `VectorValue`, записывается через `FieldValue.vector(...)`).

## 2. Создание векторного индекса (ОБЯЗАТЕЛЬНО перед первым поиском)

Firestore **не создаёт векторные индексы автоматически** — в отличие от обычных
полей, KNN-индекс над полем `embedding` нужно создать вручную через `gcloud`.
Без него `findNearest(...)` в `server/lib/vectorStore.ts` будет падать с ошибкой
`FAILED_PRECONDITION` (Firestore обычно отдаёт в тексте ошибки готовую команду
для создания индекса — это тоже надёжный способ получить точный синтаксис).

Ориентировочная команда (⚠️ **проверить флаги** перед запуском — синтаксис
`gcloud firestore indexes composite create` для vector-config менялся между
версиями gcloud CLI, сверьтесь с `gcloud firestore indexes composite create --help`
и актуальной документацией Firebase на момент деплоя):

```bash
gcloud firestore indexes composite create \
  --project=<ВАШ_PROJECT_ID> \
  --collection-group=chunks \
  --query-scope=COLLECTION_GROUP \
  --field-config=vector-config='{"dimension":"768","flat":"{}"}',field-path=embedding
```

Пояснения:
- `--collection-group=chunks` — коллекция называется `chunks` под каждым
  `users/{uid}/chunks`; `--query-scope=COLLECTION_GROUP` не обязателен, если
  запросы всегда идут к конкретной подколлекции `users/{uid}/chunks` (а не через
  `collectionGroup('chunks')` по всем пользователям сразу) — в текущей реализации
  (`vectorStore.ts`) поиск всегда идёт по `users/{uid}/chunks`, так что подойдёт
  и `--query-scope=COLLECTION`. Уточните по актуальной документации на момент
  деплоя, какой scope сейчас требует Firestore для вашего кейса.
- `dimension: 768` — должно совпадать с `EMBED_DIM` в `server/lib/embeddings.ts`.
- `flat` — тип ANN-индекса (единственный поддерживаемый Firestore на момент
  написания промпта для этой задачи); distance measure (`COSINE`/`EUCLIDEAN`/
  `DOT_PRODUCT`) задаётся не в индексе, а на уровне запроса — в коде используется
  `COSINE` (см. `distanceMeasure: 'COSINE'` в `searchChunks`).
- Создание индекса на непустой коллекции может занять время (индексация в
  фоне) — до первого построения индекса поиск будет отдавать ошибку.

Индекс создаётся **один раз на проект/базу**, не на пользователя.

## 3. Что индексируется автоматически

Начиная с этого изменения, каждая новая транскрибированная запись **автоматически**
индексируется для поиска:

1. `server/routes/processing.ts` после того как транскрипт распарсен и записан
   в Firestore (`docRef.update(...)`), вызывает:
   - `chunkTranscript(...)` (`server/lib/chunk.ts`) — режет транскрипт на куски
     ~400–600 символов + один сводный кусок из `summary`;
   - `embedTexts(...)` (`server/lib/embeddings.ts`) — считает эмбеддинги (768-мерные)
     батчами по ≤100 текстов за вызов;
   - `indexChunks(uid, chunks, vectors)` (`server/lib/vectorStore.ts`) — пишет
     чанки в `users/{uid}/chunks` батчами по ≤500 записей, с полем `embedding`
     как `FieldValue.vector(...)`.
2. Если индексация упала (сеть, квота, битый JSON и т.д.) — это **не роняет**
   сохранение самой транскрипции: ошибка ловится в отдельном `try/catch` и
   логируется через `console.warn`. Запись при этом останется без чанков в
   индексе (не будет находиться поиском), но сама транскрипция сохранится.

## 4. Бэкфилл старых записей (ОБЯЗАТЕЛЬНО спланировать перед релизом поиска)

Записи, созданные **до** этого изменения, не имеют чанков в `users/{uid}/chunks`
и не будут находиться через `/api/ai/search`, пока не будут проиндексированы
задним числом. Реализация бэкфилла НЕ входит в эту задачу, но вот план:

- **Разовый скрипт/эндпоинт бэкфилла** (например `server/routes/backfill.ts`,
  защищённый `requireAuth`, либо отдельный CLI-скрипт с service account):
  1. Пройтись по `users/{uid}/recordings` (для конкретного `uid` или для всех
     пользователей, если скрипт админский и не завязан на HTTP-запрос одного юзера).
  2. Для каждой записи с непустым `transcript` — вызвать тот же путь, что и в
     `processing.ts`: `chunkTranscript(transcript, meta, summary)` →
     `embedTexts(texts)` → `indexChunks(uid, chunks, vectors)`.
  3. Перед индексацией стоит проверить `deleteChunksForRecording(uid, recordingId)`,
     чтобы не задублировать чанки при повторном запуске бэкфилла (idempotency).
  4. Учитывать rate limits Gemini Embedding API — гнать бэкфилл пачками с
     паузами, а не разом по всем пользователям.
- Как временный маркер прогресса можно добавить полю записи что-то вроде
  `ragIndexedAt` (не входит в текущую задачу, но упрощает повторный запуск
  бэкфилла только для ещё не проиндексированных записей).

## 5. Права доступа (Firestore Rules)

`firestore.rules` **не менялись** и не должны — коллекция `users/{uid}/chunks`
подпадает под финальное правило `match /{document=**} { allow read, write: if false; }`,
т.е. клиент никогда не может читать/писать чанки напрямую. Все операции с
чанками идут только через `firebase-admin` на сервере (`vectorStore.ts`),
который полностью обходит Security Rules — это ожидаемо и безопасно, поскольку
единственная точка доступа к чанкам с клиента — эндпоинт `POST /api/ai/search`
за `requireAuth`, который сам решает, чьи чанки искать (`uid` берётся из
проверенного токена, а не из тела запроса).

## 6. Удаление чанков при удалении записи — ✅ СДЕЛАНО

Реализовано и подключено:

- Сервер: `POST /api/ai/chunks/delete` (за `requireAuth`, тело `{ recordingId }`)
  вызывает `deleteChunksForRecording(uid, recordingId)` — `uid` берётся из
  проверенного токена, чужие чанки удалить нельзя.
- Клиент: `deleteRecordingChunks(recordingId)` (`src/lib/api.ts`) вызывается в
  `removeRecording` (`src/App.tsx`) рядом с удалением аудио из R2.

Без этого поиск продолжал бы находить уже удалённые записи, а чанки копились бы
в Firestore. Ошибки удаления чанков намеренно не роняют удаление самой записи.
